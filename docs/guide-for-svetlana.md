## 1. Что вообще такое этот сайт

**Адрес**: `https://svetik-design.pages.dev` (пока временный, до подключения
своего домена — раздел 8–9).

**Что показывает**:

- Главная: короткое представление + 5 последних постов из твоего блог-канала
  `@Golovina_design_ambersoftloft`.
- `/works` — сетка проектов (сейчас 4 концепта).
- `/works/<имя-проекта>` — отдельная страница проекта с галереей.
- `/about` — о тебе.
- `/services` — три услуги (полный проект, авторский надзор, консультация).
- `/contact` — контакты и форма заявки.

**Где живёт сайт**:

- Контент (тексты, описания проектов, контакты) — в репозитории GitHub
  `pavloboss87-stack/svetik-design`, в папке `src/content/`. Файлы в формате
  Markdown (.md) и JSON. Когда ты сохраняешь правку в админке — изменение
  попадает в GitHub.
- Хостинг — Cloudflare Pages. Когда в GitHub появляется новый коммит, Pages
  автоматически пересобирает сайт и публикует через 2–4 минуты.
- Фото проектов — в той же папке `src/assets/projects/<имя-проекта>/`.
  Заливаются через админку.

**Что важно понимать**:

- Сайт статический. Это значит — он быстрый, бесплатный (Cloudflare free tier),
  и не сломается от наплыва посетителей. Минус: любое изменение требует
  пересборки (2–4 минуты). На практике это незаметно — ты сохраняешь правку
  и через несколько минут видишь её на сайте.
- Контент полностью твой. Ты в любой момент можешь забрать всё содержимое
  (это всё-таки просто файлы в GitHub) и переехать на любой другой хостинг.
  Никакого vendor lock-in.

**Что я (брат) делаю, что — ты**:

- **Я** — код, структура, дизайн (этап ep02), CI/CD, инфраструктура. Если
  сайт сломался технически — это ко мне.
- **Ты** — контент: добавить проект, обновить услугу, поменять контакты,
  ответить на заявку. Через админку, без меня.

## 2. Админка `/admin/`

Админка — это веб-интерфейс на адресе `https://svetik-design.pages.dev/admin/`.
В ней ты редактируешь все тексты, проекты, услуги и контакты. Никакого
программирования.

### 2.1 Первый вход

1. Открой `https://svetik-design.pages.dev/admin/` в браузере.
2. Жмёшь «Login with GitHub». Откроется маленькое окно GitHub.
3. Вводишь свой логин и пароль GitHub (если их ещё нет — сначала
   [регистрируйся на github.com](https://github.com), потом пиши мне, я добавлю
   тебя в репозиторий как collaborator. Без этого админка не пустит — она
   проверяет, что ты в списке).
4. GitHub спросит «Authorize svetik-design admin» — нажми Authorize.
5. Окошко закроется, ты вернёшься в админку. Появится список разделов слева.

### 2.2 Что есть в админке (9 разделов)

| Раздел                  | Что внутри                                                    |
| ----------------------- | ------------------------------------------------------------- |
| Проекты                 | 4 проекта (project-01..04). CRUD: новый, править, удалить.    |
| Услуги                  | 3 услуги (full-design, supervision, consulting).              |
| Главная — Hero          | Один файл `hero.md`. Короткое представление сверху главной.   |
| О себе                  | Один файл `about.md`. Текст + фото автора (`authorPhoto`).    |
| Услуги — вступление     | Короткий текст перед карточками услуг на `/services`.         |
| Контакты — вступление   | Короткий текст перед каналами связи на `/contact`.            |
| Политика ПДн            | Текст политики обработки персональных данных (152-ФЗ).        |
| Контакты (данные)       | 4 канала связи + email. Меняешь — на сайте обновляется.       |
| SEO (настройки)         | Имя сайта, URL, картинка для соцсетей по умолчанию.           |

### 2.3 Как добавить новый проект

1. Слева — раздел **Проекты** → жми **New Проекты**.
2. Заполни поля:
   - **Slug** (адрес страницы) — латиницей через дефис, например `kvartira-zarechye`. Это станет URL `/works/kvartira-zarechye/`.
   - **Заголовок** — как проект называется на странице (например, «Квартира на Заречной»).
   - **Год** — год работы над проектом (от 2020 до 2030).
   - **Тип** — выбираешь из списка: «Квартира», «Дом», «Студия», «Коммерческое помещение».
   - **Площадь (м²)** — число.
   - **Локация** — например, «Москва» (опциональное поле, можно оставить пустым).
   - **Описание (короткое)** — до 200 символов. Это саммари в карточке на `/works`.
   - **Обложка** — нажимаешь, выбираешь файл с компьютера. Это главное фото, оно же первое в галерее.
   - **Галерея** — минимум 3 фото. Перетаскиваешь файлы или жмёшь «Choose an image» по одному. Порядок можно менять перетаскиванием.
   - **Опубликован** — оставь «галка стоит» (по умолчанию).
   - **Концепт-проект** — «галка стоит» означает, что на сайте у проекта появится плашка «Концепт-проект · фото в обработке». Сейчас все 4 проекта помечены так, потому что у фото ещё есть watermark рендер-сервиса. Когда ты заменишь все фото проекта на отретушированные — снимаешь галку.
   - **Сортировка (order)** — число. Чем больше, тем выше в списке `/works`. Если хочешь, чтобы новый проект был сверху — поставь число больше, чем у текущего верхнего (сейчас project-01 = 4, поэтому новому ставь 5).
   - **Текст про концепт** (внизу, большое поле Markdown) — описание проекта свободным языком. См. раздел 5 «Копирайт».
3. Жмёшь **Save**. Создаётся PR (Pull Request) в GitHub. У PR появляется автоматическая проверка CI (2–3 минуты). Если CI зелёный — открываешь PR в GitHub и жмёшь Merge. После merge через 2–4 минуты проект появляется на сайте.

> **Про editorial workflow**: админка настроена так, что сохранение создаёт
> PR, а не сразу пушит в `main`. Это страховка — если ты случайно сломаешь
> схему (например, забудешь обязательное поле), CI это поймает до публикации.
> Чтобы изменение появилось на сайте, нужно открыть PR в GitHub и нажать
> «Merge pull request».

### 2.4 Как править существующий проект

1. Слева — **Проекты** → кликаешь по нужному.
2. Меняешь что нужно.
3. **Save** → PR → CI → Merge.

То же самое с услугами, текстами страниц и контактами.

### 2.5 Как удалить проект

1. Открываешь проект.
2. Сверху справа кнопка **Delete entry**.
3. PR → CI → Merge.

Файл проекта (`src/content/projects/<slug>.md`) удалится. На сайте — через
2–4 минуты после merge. Папка с фото в `src/assets/projects/<slug>/`
остаётся в репо (Decap так устроен) — это не страшно, просто лежит как
архив. Если хочется удалить и фото — пиши мне, удалю руками одним коммитом.

### 2.6 Как поменять контакты (телефон, Telegram, email)

Самое частое — что-то поменялось в связках.

1. Слева — **Контакты (данные)**.
2. Меняешь поля:
   - **Telegram (личный)** — `label` отображается, `url` ведёт куда надо.
   - **Telegram (блог)** — то же.
   - **VK** — то же.
   - **Max** — мессенджер max.ru.
   - **Email** — текущий: `example@email.com` (заглушка). Меняешь на боевой
     перед публичным запуском (это в чек-листе активации).
   - **Primary** — отметка «основной канал», подсвечивается на сайте.
3. **Save** → PR → Merge.

### 2.7 Куда смотреть, если что-то не получилось

- **PR висит с красным CI** — открой PR в GitHub, прокрути вниз, в блоке
  «Checks» нажми на красный пункт. Там написано, что упало. Скорее всего —
  забыла обязательное поле или поставила слишком длинный текст в поле с
  лимитом. Возвращайся в админку, правь, **Save** заново — это обновит тот же PR.
- **Сохранение в админке выдаёт ошибку** — скриншот ошибки в Telegram мне,
  разберусь.
- **Залогинилась, но коллекций нет / админка пустая** — выйди (`Logout`
  сверху справа), залогинься заново. Если не помогло — пиши мне.

### 2.8 Скриншоты

> Я (брат) обещал положить скриншоты в `docs/guide-screenshots/`. Их пока
> нет — заполню в момент финального деплоя (T31). Если они тебе нужны
> прямо сейчас и не дождёшься — открой админку, всё устроено по логике
> «слева список разделов, внутри — список записей, клик по записи открывает
> форму». Кнопки **New**, **Save**, **Delete entry**, **Publish** видны
> сразу.

## 3. Ретушь watermark на фото проектов

Сейчас у фото 4 концептов watermark рендер-сервиса. Сайт показывает их
с явной плашкой «Концепт-проект · фото в обработке» — это честная подача
(никто никого не вводит в заблуждение).

Когда у тебя дойдут руки до ретуши — ниже два инструмента и порядок действий.

### 3.1 IOPaint (бесплатно, локально, batch)

[IOPaint](https://github.com/Sanster/IOPaint) — десктоп-приложение для
очистки фото от объектов через inpainting. Бесплатное, работает локально
без интернета.

**Установка** (один раз):

1. Скачай [Python 3.10+](https://www.python.org/downloads/) если ещё нет.
2. Открой Terminal (Mac) или PowerShell (Windows).
3. Введи команду:

    ```
    pip install iopaint
    ```

4. Подожди 2–5 минут.

**Использование**:

1. Открой Terminal/PowerShell.
2. Введи:

    ```
    iopaint start --model=lama --port=8080
    ```

3. Открой в браузере `http://localhost:8080`.
4. Загружай фото по одному, кисточкой закрашиваешь watermark, жмёшь Run.
5. Скачиваешь результат через «Download».

**Плюсы**: бесплатно, batch (можно много за вечер), результат хороший
на простых watermark.

**Минусы**: интерфейс на английском, для сложных watermark (полупрозрачные,
наложенные на узор) результат бывает грязным — тогда переходи на пункт 3.2.

### 3.2 Cleanup.pictures (платно, сложные кадры)

[cleanup.pictures](https://cleanup.pictures) — онлайн-сервис, веб-интерфейс.
Хорошо справляется со сложными случаями, где IOPaint оставляет артефакты.

- Бесплатная версия: до 720p, один файл.
- Платная: до $9/месяц, без ограничений.

Используй только для тех 1–2 кадров, где IOPaint не справился.

### 3.3 Порядок действий по проекту

1. Открой все исходники одного проекта (например, project-02 — 3 фото).
2. Прогони через IOPaint всё, что получается.
3. Что не получилось — через cleanup.pictures.
4. Сохрани финальные файлы локально (например, в папку
   `~/Projects/svetik-design-photos/project-02-ready/`).
5. Идёшь в админку (раздел 2.4) → правишь project-02 → удаляешь старые фото
   из галереи (крестик на превью) → загружаешь новые через **Choose an image**.
6. Снимаешь галку «Концепт-проект» — плашка с сайта пропадёт.
7. **Save** → PR → CI → Merge.

Через 2–4 минуты на сайте у этого проекта плашки уже нет, фото чистые.

## 4. Подготовка фото перед заливкой

Чтобы сайт быстро грузился и хорошо выглядел — несколько правил для фото
перед заливкой в админку.

### 4.1 ⚠️ iPhone HEIC — обязательно конвертировать в JPEG

**iPhone по умолчанию сохраняет фото в формате HEIC. Сайт его не показывает.
Если зальёшь HEIC напрямую — Astro не сможет собрать билд, PR будет красным.**

Два способа исправить:

**Вариант А: настроить iPhone, чтобы он сразу делал JPEG** (рекомендую):

1. На iPhone: **Настройки → Камера → Форматы → Наиболее совместимый**.
2. С этого момента все новые фото — в JPEG. Старые HEIC так и останутся
   HEIC, их нужно конвертировать через вариант Б.

**Вариант Б: конвертировать готовые HEIC**:

- **На Mac**: открой HEIC в Preview/Просмотр → **Файл → Экспортировать → Формат JPEG → Сохранить**.
- **На Windows**: установи [HEIC-конвертер](https://www.microsoft.com/store/productid/9PMMSR1CGPWG) из Microsoft Store, потом правый клик по файлу → Convert to JPEG.
- **Онлайн (если конвертеров нет)**: [heictojpg.com](https://heictojpg.com) — drop, скачать. Не для секретных фото (онлайн-сервис).

### 4.2 Размер и разрешение

- **Минимум**: 1200×900 пикселей (короткая сторона ≥900).
- **Оптимум**: 1600×1200 или 2000×1500.
- **Максимум**: до 4000 пикселей по длинной стороне. Больше — лишний вес,
  Astro всё равно сожмёт до 1200 для галереи.

Не нужно заранее ничего ресайзить — Astro делает 3 размера (`400`, `600`,
`800` пикселей для карточек на `/works`, и `600`, `900`, `1200` для галереи)
автоматически.

### 4.3 Кроп

- **Обложка проекта** (`cover`) — горизонтальное соотношение 4:3 или 3:2.
  В карточке `/works` отображается с обрезкой по высоте — лучше композиция
  «в центре кадра, по краям не критично».
- **Галерея** — любое соотношение. Сайт показывает все фото с пропорцией
  4:3 (вписывает в кадр). Вертикальные тоже работают — просто будут с
  пустыми полями сверху/снизу.

### 4.4 Имена файлов

Англо-цифровые, без пробелов и кириллицы. Например: `cover.jpg`, `01.jpg`,
`02.jpg`. Decap при загрузке покажет имя — можешь его переименовать в
интерфейсе перед сохранением.

Кириллица в имени иногда ломает image-pipeline Astro на сборке в облаке
(Linux чувствителен к UTF-8 именам файлов в URL'ах).

### 4.5 Ориентация

iPhone иногда пишет в EXIF «портрет» для фактически горизонтального снимка
(если телефон был в руках под углом). Если на превью в Decap фото
повёрнуто — открой в Preview/Просмотр → **Инструменты → Повернуть** → Сохрани
→ перезалей.

### 4.6 Чек-лист перед сохранением проекта

- [ ] Все фото — JPEG, не HEIC.
- [ ] Имена файлов латиницей.
- [ ] Минимум 3 фото в галерее (включая cover).
- [ ] Ориентация правильная (открой в браузере, проверь).
- [ ] Watermark убран (если ещё нет — оставь галку «Концепт-проект»).

## 5. Копирайт — как писать тексты для сайта

Сайт-визитка для платёжеспособной аудитории. Платёжеспособный клиент
отличает живой авторский текст от копирайтерской рутины с первой
строчки. Слабый текст здесь не нейтрален — он **отталкивает**.

### 5.1 Что **не** писать никогда

Эти штампы выдают «нейросетевой» / «копирайтерский» текст:

- «уют и функциональность»
- «стиль и комфорт»
- «ваш дом мечты»
- «создаём пространства, в которых хочется жить»
- «индивидуальный подход»
- «превращаем интерьер в произведение искусства»
- «не просто X, а Y» (риторический оборот)
- «это больше чем дизайн»
- «delivering excellence», «годы опыта», «команда профессионалов»
- любые риторические перечисления через тире

Сайт настроен так, что если эти фразы попадут в текст — CI (автопроверка)
не упадёт (это не код), но я при ревью увижу и поправлю. Лучше — сразу так
не писать.

### 5.2 Как писать про проект

Структура одного проекта (200 символов на саммари + 2–5 абзацев на текст
концепта):

1. **Что было**. Откуда стартовала, какая исходная задача. Конкретно — не
   «нужен был уютный дом», а «семья из четырёх человек, переехали в новостройку
   65 м², кухня и гостиная объединены, два окна на восток».
2. **Что хотелось**. Конкретно. Не «современный стиль», а «отказ от верхнего

## 6. Telegram-бот заявок — настройка

Сейчас форма на `/contact` отправляет заявки в **mock-режим** — Worker
возвращает «успешно», но реально никуда не пишет. Чтобы заявки начали
приходить тебе в Telegram — нужно создать бота и настроить.

> **Что я сделал заранее**: написал и задеплоил серверный код (Cloudflare
> Worker `svetik-design-submit`), который умеет принимать форму, валидировать
> и отправлять. Не хватает только секретов — токена бота и твоего chat_id.

### 6.1 Создаёшь бота через @BotFather

1. Открой в Telegram бот [@BotFather](https://t.me/BotFather).
2. Команда `/newbot`.
3. Имя бота — любое, например `Svetik Design заявки`.
4. Username (логин) бота — должен заканчиваться на `bot`, например
   `svetik_design_inbox_bot`. Если занят — придумай другой.
5. BotFather ответит: «Use this token to access the HTTP API:
   `1234567890:AAExAmPle…`». Это **TELEGRAM_BOT_TOKEN**. Сохрани в
   надёжное место (например, в Apple Notes под паролем). **Никому не показывай**
   — кто угодно с этим токеном сможет отправлять сообщения от твоего бота.

### 6.2 Получаешь свой chat_id

1. Открой в Telegram бот [@userinfobot](https://t.me/userinfobot).
2. Команда `/start`.
3. Бот ответит: «Id: 123456789». Это **OWNER_CHAT_ID** — твой
   персональный идентификатор в Telegram.

### 6.3 Запускаешь своего бота, чтобы он мог тебе писать

1. Найди в поиске Telegram своего бота (username из 6.1).
2. Жми **Start** или просто пиши ему `привет`. Один раз. Без этого Telegram
   не разрешает боту инициировать диалог.

### 6.4 Прописываешь секреты в Worker'е

Это техническая часть. Если не получится — пиши мне, я сделаю.
Самой можно так:

1. Скачай Cloudflare wrangler CLI один раз: `npm install -g wrangler`
   (нужен Node.js — пиши, объясню установку отдельно).
2. В терминале:

   ```
   wrangler login
   ```

   Откроется браузер — логинься в свой аккаунт Cloudflare.

3. Перейди в папку проекта:

   ```
   cd C:\Proj\Svetik-design\workers\submit
   ```

4. Введи:

   ```
   wrangler secret put TELEGRAM_BOT_TOKEN
   ```

   Терминал спросит значение — вставь токен бота из 6.1, Enter.

5. Введи:

   ```
   wrangler secret put OWNER_CHAT_ID
   ```

   Вставь chat_id из 6.2.

6. Передеплой:

   ```
   wrangler deploy
   ```

### 6.5 Проверка

1. Открой `https://svetik-design.pages.dev/contact`.
2. Заполни форму своим именем, контактом, сообщением «тестовая заявка».
3. Поставь галочку «Согласен с политикой обработки персональных данных».
4. Жми **Отправить**.
5. Через 2–5 секунд тебе в Telegram придёт сообщение от твоего бота с
   текстом заявки.

Если не пришло — пиши мне.

### 6.6 Безопасность

- Если кто-то узнает TELEGRAM_BOT_TOKEN — он сможет писать сообщения от
  имени бота. Чтобы перезапустить токен: в @BotFather → `/mybots` → выбери
  бота → **API Token → Revoke current token**. После — повтори 6.4 с
  новым токеном.
- Бот может писать только тебе (потому что OWNER_CHAT_ID — твой). Чтобы
  бот писал в группу — нужны другие настройки, пиши мне отдельно.

## 7. Email-копия заявок — настройка

В дополнение к Telegram — копия каждой заявки приходит на твою почту. Я
заложил инфраструктуру под Яндекс.Почту (SMTP), но **реальная отправка
email пока не реализована** — это задача следующего этапа (ep03), когда
ты решишь, нужна ли вообще email-копия (Telegram-уведомления могут
оказаться достаточными).

Если решишь, что нужна — план такой:

### 7.1 Создай отдельную рабочую почту

Не используй личный email. Заведи отдельный, чтобы:

- Заявки не терялись среди других писем.
- Если что-то случится с твоим личным аккаунтом — рабочие заявки не
  потеряются.

Рекомендую: `golovina.svetlana.design@yandex.ru` или похожее. На
[yandex.ru](https://yandex.ru) → регистрация → новый аккаунт.

### 7.2 Создай пароль приложения для SMTP

Яндекс не разрешает использовать пароль от аккаунта в почтовых программах —
нужен отдельный «пароль приложения».

1. Залогинься в Яндекс.Почту.
2. Открой [id.yandex.ru/security](https://id.yandex.ru/security).
3. **Пароли приложений** → **Создать пароль** → выбери «Почта».
4. Яндекс покажет пароль вида `xxxxxxxxxxxxxxxx`. **Сохрани его** — после
   закрытия страницы он больше не покажется.

### 7.3 Когда мы будем подключать (ep03)

- Я заложу SMTP-отправку в Worker (с MailChannels или Яндекс SMTP).
- Ты прокинешь `SMTP_USER` (твой email) и `SMTP_PASS` (пароль приложения)
  через `wrangler secret put`, аналогично 6.4.
- Сделаем тестовую заявку — придёт и в Telegram, и в почту.

Пока — пиши мне, когда будешь готова добавить email-канал. На этапе ep01
обходимся Telegram'ом.

## 8. Свой домен — что это и зачем

Сейчас сайт живёт на `svetik-design.pages.dev`. Это технический URL Cloudflare Pages.
Работает, но выглядит как «временный».

Для публичного запуска лучше свой домен — например, `golovinadesign.ru`. Тогда:

- На визитке/в Instagram/в постах ты пишешь «golovinadesign.ru», не
  «svetik-design.pages.dev».
- Email на этом домене (`hello@golovinadesign.ru`) выглядит профессиональнее.
- Сайт «принадлежит» тебе на уровне поисковиков — если когда-то решишь
  сменить хостинг (см. раздел 10), URL у клиентов в закладках остаётся
  тот же.

### 8.1 Где купить

В РФ актуально (на середину 2026) — [reg.ru](https://reg.ru).

- **.ru** — ~199 ₽/год, российская зона.
- **.online** / **.studio** / **.design** — ~700–1200 ₽/год, нишевые.
- **.com** — ~900 ₽/год, международный, но в РФ иногда хуже резолвится
  через провайдерскую DNS.

Рекомендую **.ru**. Дешёвый, надёжный, для аудитории «дизайнер интерьеров
в Москве» — самый понятный сигнал.

### 8.2 Как выбрать имя

- Простое, легко произнести по телефону.
- Без цифр и дефисов (пишут с ошибками).
- Желательно совпадает с тем, что ты уже используешь — `golovinadesign` если
  блог `Golovina_design_ambersoftloft` (можно сократить до основной части).
- Проверь на [reg.ru](https://reg.ru) поиск — свободен или нет.

Варианты, которые я бы рассмотрел: `golovinadesign.ru`, `golovina-design.ru`,
`svgodesign.ru` (совпадает с `@svgodesign` в Telegram).

### 8.3 Покупка

1. Регистрируешься на reg.ru.
2. В поиске вводишь желаемое имя.
3. Если свободен — **Купить**, оплачиваешь.
4. **Не покупай ни хостинг, ни почту в дополнение** — для сайта это не
   нужно (хостинг у нас Cloudflare, бесплатно; почту делает Яндекс
   отдельно).
5. Подтверждаешь данные администратора по email.

После покупки домен «твой» — но к сайту его ещё надо подключить (раздел 9).

## 9. Подключение домена к сайту

После покупки домена в reg.ru — настройка занимает ~30 минут + ~24 часа
ожидания пока DNS-записи распространятся по интернету.

### 9.1 Добавь домен в Cloudflare Pages

1. Логинься в [dash.cloudflare.com](https://dash.cloudflare.com).
2. **Workers & Pages → svetik-design → Custom domains**.
3. **Set up a custom domain** → введи свой домен (например, `golovinadesign.ru`).
4. **Continue → Activate domain**.
5. Cloudflare покажет 2 записи DNS, которые нужно прописать у регистратора
   (в reg.ru). Скриншот их или скопируй (это `CNAME` или `A`-записи).

### 9.2 Пропиши DNS в reg.ru

1. Логинься в [reg.ru](https://reg.ru) → **Мои домены** → выбираешь свой.
2. **Управление DNS-серверами** или **Управление зоной**.
3. Удали все существующие записи (если есть).
4. Добавь те 2 записи, что Cloudflare выдал в 9.1.
5. **Сохранить**.

> Альтернативный путь: вместо точечного DNS можно перевести **весь домен**
> на Cloudflare nameservers (это даёт CDN-преимущества). В reg.ru — поменять
> NS-серверы на те, что Cloudflare предложит при добавлении домена в
> Cloudflare как «Add site». Но это сложнее для первого раза. Точечный
> DNS — проще и достаточно.

### 9.3 Подожди

DNS-записи распространяются от 5 минут до 24 часов. Чаще всего — час-два.
Проверка: открой `https://golovinadesign.ru` в браузере — должен
отрисоваться сайт. Если 24 часа прошло, а сайта нет — пиши мне.

### 9.4 Обнови URL в настройках сайта

После того как `https://golovinadesign.ru` работает:

1. В админке открой **SEO (настройки)** → **Site URL** → меняй с
   `https://svetik-design.pages.dev` на `https://golovinadesign.ru`.
2. **Save** → PR → CI → Merge.
3. Скажи мне — я обновлю несколько файлов конфигурации, которые админка
   не покрывает (`_headers` CSP, OAuth Worker callback URL, Submit
   Worker CORS), и пересоберу OAuth Worker на новый Authorization callback URL.

После этого старый `svetik-design.pages.dev` продолжит работать (Cloudflare
оставляет оба URL), но Google и поисковики увидят `golovinadesign.ru` как
канонический.

### 9.5 Проверка из РФ

Из-за политики РКН-РФ некоторые домены/хостинги могут резолвиться нестабильно
у российских провайдеров. Cloudflare периодически попадает под блокировки
по IP, но `*.pages.dev` и кастомные домены через Cloudflare обычно работают.

Проверь сайт из:

- Дома (Ростелеком/МТС/Билайн).
- Через мобильный 4G.
- У знакомого в другом городе.

Если откуда-то не открывается — пиши мне. План B — раздел 10.

## 10. Резервный план — переезд на Timeweb

На случай, если Cloudflare заблокируют в РФ или регулятор начнёт системно
давить — переезжаем на [Timeweb Cloud](https://timeweb.cloud) (российский
хостинг с поддержкой статических сайтов).

### 10.1 Что не поменяется

- Контент твой, ничего не теряется. Это всё файлы в GitHub.
- Админка — тоже работает, потому что она тоже файлы в GitHub
  (`public/admin/`).
- Domain `golovinadesign.ru` остаётся твоим (купленным у reg.ru), просто
  DNS-записи переведутся с Cloudflare на Timeweb.

### 10.2 Что нужно сделать (займёт ~2 часа)

- **Я** (брат):
  1. Регистрирую тебе Timeweb Cloud — раздел «Apps» (для статических сайтов
     с GitHub).
  2. Подключаю репозиторий, настраиваю build command (`pnpm install --frozen-lockfile && pnpm build`), output `dist/`.
  3. Деплою — получаю URL вида `<name>.timeweb-cloud.ru`.
  4. Прохожу заново через CSP в `public/_headers` (там зашиты URL'ы Workers).
  5. Workers (OAuth, Submit) переношу либо на Timeweb Cloud Functions, либо
     оставляю на Cloudflare Workers (они не блокированы, в отличие от Pages).
  6. Перевожу DNS-записи с Cloudflare на Timeweb.
- **Ты**:
  1. Платишь Timeweb (от 200 ₽/месяц для статики).
  2. Скриншот build-логов мне для архива.

### 10.3 Когда переезжать

- **НЕ переезжать заранее**. Cloudflare пока работает, экономия очевидная
  (бесплатно vs 200 ₽/мес).
- **Переезжать, когда**: 2 недели подряд клиенты говорят «не открывается»;
  Google Search Console показывает падение трафика; или регулятор официально
  внёс CF в реестр блокировок.

## 11. Branch protection в GitHub — что это и зачем

В GitHub-репозитории включена настройка «branch protection» на ветке `main`.
Что это значит на практике:

- **Никто** (включая меня) не может запушить изменения напрямую в `main` — только через Pull Request.
- Каждый PR проходит **5 автоматических проверок** (CI): `lint`, `typecheck`, `test`, `lighthouse`, `e2e`. Все 5 должны быть зелёными.
- Если хоть одна красная — кнопка «Merge pull request» в GitHub серая и не работает.

**Зачем**:

- Защита Принципа 3 из Constitution — сайт не уходит в продакшен с упавшим
  Lighthouse score (медленный, недоступный для AT, плохой SEO). Без этой
  настройки правило «95+» — пожелание; с настройкой — невозможно нарушить.

**Что это значит для тебя**:

- Когда ты в админке нажимаешь **Save** — открывается PR, CI запускается.
  Если зелёный — открываешь PR в GitHub и жмёшь Merge. Защита автоматически
  подтверждает «можно мержить».
- Если CI красный — ничего страшного, переоткрываешь запись в админке,
  правишь, сохраняешь заново (PR обновится). Защита просто не даст
  опубликовать сломанное.

**Что делать, если что-то сломалось**:

- CI красный на всех PR подряд, даже на мелких правках текста — значит
  что-то сломалось на инфраструктурном уровне (обновился API GitHub /
  Cloudflare поменял что-то). Пиши мне.
- **Никогда не выключай защиту сама**. Если очень-очень нужно — пиши мне,
  включу обратно сразу после фикса.
