## Codebase Patterns

<!-- Reusable patterns discovered during implementation. Keep this section at the top. -->

- **pnpm install on Windows without admin**: `corepack enable` writes shim files into `C:\Program Files\nodejs` and fails with EPERM under a non-elevated shell. Use `npm install -g pnpm@<major>` instead — it writes to `%APPDATA%\npm`, which is user-writable. Refresh `$env:Path` from `Machine` + `User` env vars in the same session before invoking `pnpm`.
- **`pnpm create astro` in a non-empty directory**: the interactive prompt blocks even with `--yes`. Workaround: scaffold into a sibling temp dir (`_astro-tmp/`), move files into the project root, delete the temp dir. Existing tracked source folders (`Portfolio/`, `Contacts/`, `Diplom/`, `docs/`, `CLAUDE.md`) survive intact.
- **Astro CLI version drift vs. plan**: the plan refers to "Astro 5", actual scaffold produces Astro 6.x. Both share the `content collections` + `loader` API; downstream task specs (especially T07 mentioning "Astro 5 — `loader` API") still apply.
- **Tailwind 4 + Astro**: no `tailwind.config.mjs`. Wire-up is (a) `@tailwindcss/vite` plugin in `astro.config.mjs#vite.plugins`, (b) `@import 'tailwindcss';` in a CSS entry file, (c) import that CSS from each page or `Layout.astro`. Theme tokens go inside the CSS file via `@theme { ... }` directives (planned for ep02). No `content` glob — auto-scan via Vite.
- **Astro routing skip rule**: files or directories prefixed with `_` under `src/pages/` are excluded from the route tree (so they are not built). Useful for fragments and partials. For one-off scratch/test pages that need to be built, prefer a non-underscored name (e.g. `temp-foo.astro`) and delete after verification.
- **Sharp as explicit devDep**: even though `sharp` ships transitively with Astro (for image optimization), Node ESM resolution from a sibling script in `scripts/` cannot import it unless it's a top-level dependency. Add `sharp` to `devDependencies` if you need to use it directly in build helpers.
- **Astro 6 + Zod 4 imports for content schemas**: `import { z } from 'astro:content'` is deprecated in Astro 6 (removal flagged for Astro 7) — use `import { z } from 'astro/zod'` together with `import { defineCollection } from 'astro:content'`. In Zod 4 the instance methods `.url()` / `.email()` are deprecated in favour of the top-level constructors `z.url()` / `z.email()`. Both surface as TS hints (severity `hint`, not `warning` or `error`), so `pnpm typecheck` stays green either way — using the new forms keeps the file at 0/0/0.
- **Single-entry settings as a `glob` JSON collection + `z.union`**: when you want `getEntry('settings', 'contacts')` and `getEntry('settings', 'seo')` (one collection, multiple distinct shapes from separate JSON files), use `glob({ pattern: '**/*.json', base: './src/content/settings' })` with `schema: z.union([contactsSchema, seoSchema])`. Each `.json` file becomes one entry whose `id` is its filename without extension. The schemas must have non-overlapping required keys so the union discriminates correctly. Cleaner than per-file `file()` collections — keeps a single collection key and matches Decap's «one file-collection per settings file» layout from plan Step 5.
- **`image()` schema paths are relative to the markdown file, not the project root**: frontmatter like `cover: ../../assets/projects/project-01/01.jpg` resolves when the .md lives at `src/content/projects/project-01.md` and the image at `src/assets/projects/project-01/01.jpg`. Reusing the same path string for `cover` and the first entry of `gallery` does NOT duplicate bytes — Astro+Vite dedupes by resolved path, so `cover === gallery[0]` is a valid pattern for seeding when source has few photos.
- **One-shot placeholder image generators**: `scripts/generate-*-placeholder.mjs` (one per use case) using `sharp(Buffer.from(svg)).jpeg(...).toFile(...)` is the established pattern. Used for apple-touch-icon (T04), `project-01/03.jpg` (T08), and `about/placeholder.jpg` (T09). Idempotent, run once to seed; outputs land under `src/assets/...` and are versioned in git.
- **Astro dev server does NOT auto-serve `public/<dir>/index.html` for trailing-slash directory requests**. `/admin/` and `/admin` both return 404 in `pnpm dev`; only `/admin/index.html` works. `pnpm preview` (production-like static) and CF Pages resolve trailing-slash to `index.html` natively. Fix used in T11: dev-only Vite plugin in `astro.config.mjs#vite.plugins` that rewrites `req.url` for known static-dir paths inside `configureServer`. Pattern (copy-paste for next case): plain object with `name` and `configureServer(server) { server.middlewares.use((req, _res, next) => { if (req.url === '/foo' || req.url === '/foo/') req.url = '/foo/index.html'; next(); }); }`. Annotate params with inline JSDoc — `vite` is not a direct dep (transitive of Astro), so `import('vite').Plugin` triggers TS 2307 in IDE. JSDoc inline `@param {{ middlewares: { use: ... } }}` is enough and avoids pulling `vite` into devDeps.
- **`astro check` only typechecks `.astro` files, NOT `.mjs`/`.ts` config files**. IDE language server (VS Code TypeScript) DOES flag `// @ts-check`'d `astro.config.mjs` with TS errors (e.g. TS 7006 implicit any), but `pnpm typecheck` walks past them. So IDE-only diagnostics on the config are real but not CI-gated. When editing config, run the actual file through `node --check` or eyeball it; don't rely on `pnpm typecheck` to catch config-level TS issues.
- **Decap CMS shell pattern (Astro + CDN-loaded Decap)**: `public/admin/index.html` is a 12-line shell with `<meta name="robots" content="noindex,nofollow">` and `<script src="https://unpkg.com/decap-cms@^3.0.0/dist/decap-cms.js"></script>`. `public/admin/config.yml` carries the entire schema. No npm dependency, no build step — `pnpm build` just copies `public/` verbatim. Decap reads config.yml at runtime over HTTP. For local UI-only smoke tests (without GitHub OAuth) use `backend: { name: test-repo }` — renders UI, lets you click into «+ Create» and inspect every field widget. Production uses `backend: { name: github, base_url: <oauth-worker-url> }` and requires the OAuth proxy Worker.
- **Decap `media_folder` paths must be relative to the entry directory, not project root**, to produce frontmatter paths compatible with Astro's `image()` schema. For folder collection `projects` (entries at `src/content/projects/<slug>.md`) with photos in `src/assets/projects/<slug>/`, set `media_folder: '../../assets/projects/{{slug}}'` and `public_folder: '../../assets/projects/{{slug}}'` AT THE COLLECTION LEVEL (not on individual image fields). Decap then writes `cover: ../../assets/projects/<slug>/<file>.jpg` in frontmatter, which Astro `image()` resolves from the .md file path. For file collection entries (e.g. `pages/about.md`) per-file `media_folder: '../../assets/about'` does the same. If you forget this and use only global `media_folder: src/assets/uploads`, Decap writes absolute paths like `/src/assets/uploads/foo.jpg` and Astro's `image()` schema rejects them at build.
- **Two synced model descriptions (Zod + Decap yaml) — accept and manage manually**: `src/content.config.ts` and `public/admin/config.yml` describe the same content shape in two languages. No code-gen (Key Decision in plan.md Step 10). When adding or renaming a field, edit both. Mismatches surface in two ways: (a) Decap saves a file that fails Zod on next `pnpm build` — caught loud in CI; (b) Decap UI lacks a widget for a Zod-required field — silent until sister tries to create an entry. Practice: keep `src/content.config.ts` open in one tab and `public/admin/config.yml` in another when changing schema; eyeball both sides before commit.
- **Decap-compatible OAuth proxy on a single Cloudflare Worker (stateless)**: ~110 lines, no KV/D1/R2 bindings, fits the Worker free tier comfortably. Two routes: `GET /auth` redirects to `github.com/login/oauth/authorize?client_id&redirect_uri=<worker-origin>/callback&scope=repo&state=<uuid>`; `GET /callback` POSTs `code` + secret to `github.com/login/oauth/access_token` (Accept: application/json), then returns an HTML popup that performs the Decap two-step handshake — wait for `authorization:github` from `window.opener`, reply with `authorization:github:success:{"token","provider":"github"}` (or `:error:<json-string>`). Pattern lives at `workers/oauth/src/index.ts` (T12). `OAUTH_CLIENT_ID` is a public var (committed in `wrangler.toml#[vars]`); `OAUTH_CLIENT_SECRET` is set only via interactive `wrangler secret put` — never in code, env files, or chat. Same shape reusable for any Git-backed CMS that follows the `decap-cms-lib-auth` protocol.
- **Cloudflare Worker as a non-Astro sibling subproject**: lives under `workers/<name>/` with its own `wrangler.toml`, `tsconfig.json` (typed via `@cloudflare/workers-types`), `src/index.ts`, and `README.md`. Not part of `pnpm build` (Astro static site only) — deploy is independent (`wrangler deploy` from the subproject dir). Root `package.json` carries `@cloudflare/workers-types` as a devDep so the subproject's tsconfig resolves it via root `node_modules`. `astro check` ignores `workers/` (only walks `.astro`); ESLint covers `.ts` there if not in `ignores` list of `eslint.config.js`. Worker URL is fed back into the static site at config-edit time (e.g. Decap's `backend.base_url`).
- **Decap CMS GitHub-OAuth popup handshake — exact strings matter**: `decap-cms-lib-auth#handshake_callback` listens for `'authorizing:<provider>'` from the popup and ECHOES the SAME literal back (NOT a renamed `'authorization:<provider>'`). The popup must then post `'authorization:<provider>:(success|error):<json-string>'` back to the opener. One missing 'i' in the popup's echo-check string means the success message never fires, the popup never closes, and the opener's Login button stays — manifesting as a perfect-looking OAuth round-trip that dead-ends in a loop. Fix landed in `workers/oauth/src/index.ts` (T12 fix, commit `0782b2c`). When testing, watch the popup's window.close() — Decap will close the popup itself once it receives the success message, but the popup also self-closes after 500ms as a fallback so a missed handshake is visible as a stuck popup rather than instant disappearance.
- **Decap with `backend.name: github` reads content from the REMOTE repo, not the local filesystem**. Even though `pnpm dev` serves `/admin/config.yml` from local `public/`, the collection entries are fetched via GitHub REST API against `repo` + `branch` in config. Symptom of forgetting to push: admin UI loads, login works, but collections render «Нет записей» / «No entries». Mitigation: always `git push origin main` before smoke-testing the admin UI for the first time, AND any time you add/rename files in `src/content/`.
- **Decap `publish_mode: editorial_workflow` branch + PR pattern**: on Save of an existing entry, Decap creates a remote branch named `cms/<collection>/<slug>` (e.g. `cms/projects/project-01`) with a single commit, and opens a draft PR against `branch:` from config. Decap never touches the local working tree — edits go straight to the remote via GitHub API. Cleanest cleanup for a test edit: `git push origin --delete cms/<collection>/<slug>` — GitHub auto-closes the corresponding PR as `closed` (unmerged), no merge-then-revert dance. The local file and main branch stay untouched.
- **Vitest at root, no pnpm-workspace, single `pnpm test` entry**: `vitest.config.ts` at repo root with `include: ['workers/**/test/**/*.test.ts', 'tests/unit/**/*.test.ts']` lets one root-level `vitest` devDep cover Astro `src/lib/` unit tests AND Cloudflare Worker tests under `workers/<name>/test/`. No per-worker `package.json` needed; the literal-but-misleading `cd workers/submit && pnpm test` from task briefs becomes `pnpm test` from root. Trade-off: tests can't be invoked from inside `workers/<name>/` directly via pnpm without setting up a workspace (overkill for one author). Document the deviation in the worker's README. Pattern lives at root `vitest.config.ts` + root `test` script in `package.json` (T23).
- **Astro check actually walks every `.ts` file in tsconfig `include`**: earlier (Session C) we noted «astro check only checks `.astro` files» — that was wrong, or at least incomplete. Astro check 0.9.9 uses the project tsconfig as-is; root `tsconfig.json` has `include: ['**/*']`, so EVERY `.ts` under the repo (including `vitest.config.ts`, `workers/submit/src/index.ts`, `workers/submit/test/index.test.ts`) goes through typecheck. CI-gated. What it still skips: `.mjs` config files (no TS-aware diagnostics there). Practical impact: when adding TS code anywhere, expect `pnpm typecheck` to lint it — write proper types from the start instead of `as any` casts. To exclude a tree from typecheck without breaking ESLint, add it to tsconfig `exclude` (not yet needed in this repo).
- **In-memory per-isolate rate-limit `Map` for atomic concurrent burst counting**: a plain module-level `Map<ip, {count, resetAt}>` with a sync `incrementRateLimit()` function gives atomic counting WITHIN one CF Worker isolate — JS is single-threaded, so `Promise.all(N)` calls all execute the sync increment before any of them hits an await. Cheap, no KV/D1/DO, ~15 lines. Cost: cross-isolate burst (an attacker hitting multiple CF POPs) can exceed N×limit, which is acceptable for low-traffic personal sites. Trade-off explicitly documented in `workers/submit/src/index.ts` and `workers/submit/README.md` § «Архитектурные заметки» (T23). When upgrading to true global counters, swap the Map for a Durable Object behind the same `incrementRateLimit` signature — call sites unchanged. Tested via `Promise.all(6)` returning exactly one 429.
- **Honeypot drop counts toward rate-limit + happens before validation**: anti-bot pattern in form workers. Sequence: (a) parse JSON, (b) check honeypot field (e.g. `website`) — if filled, return 200 silently with a distinct log marker like `[honeypot-drop]` (NOT `[mock]` — keep markers grep-distinct), (c) only THEN validate the real fields. Rationale: a bot probing the field name learns nothing from a 400 response; and the honeypot path MUST still increment the rate-limit counter, otherwise bots can hammer the endpoint and learn structure via timing. Tested explicitly in `workers/submit/test/index.test.ts` (T23).
- **Mock-mode trigger via AND of required secrets, asymmetric with real-mode**: contact-form/integration workers run in mock-mode (return 200 + `{mock: true}` + warn log) if ANY required secret is unset; real-mode requires ALL to be set. Pattern: `isMockMode = !TOK || !CHAT || !SMTP_USER || !SMTP_PASS`. Lets you deploy the worker early (T23) before the sister/owner has actually created the bot, mailbox, etc., and the form still demos-as-if-working. Real activation later is purely a secrets update (`wrangler secret put ...`), no code change. Asymmetric vs «mock-mode = no secrets at all» — partial config (only 1 of 4 secrets) is treated as mock too, preventing accidental half-broken sends.
- **HTML-escape user input before Telegram `parse_mode: 'HTML'`**: if you send a TG message with `parse_mode: 'HTML'`, any literal `<`, `>`, or `&` in user-supplied fields breaks Telegram's parser (400 or weird rendering). Minimal `escapeHtml(s)` = three sequential replaces (`&` → `&amp;`, then `<` → `&lt;`, then `>` → `&gt;`). The order matters: `&` first or you double-escape entities. Lives in `workers/submit/src/index.ts` (T23). Test asserts `<script>` literal is rendered as `&lt;script&gt;` and `& friends` as `&amp; friends`.
